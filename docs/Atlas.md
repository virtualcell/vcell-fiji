# Problem
Users of VCell are able to create a simulated model, but exporting and analyzing data from these models
is not intuitive and can be laborious.

If the generated models can be viewed within ImageJ, then the tools already present within ImageJ itself can
be used to process the image and export data easily.

# Potential Solutions
- Big Data Server Vanilla
  - Pros
    - Easy to set up immediately with the big data server java application.
  - Cons
    - Can only open images within the big data viewer application, which can be limiting with its 
    processing capabilities

- Custom Pipe Plugin from big data viewer
  - Customize some pre-existing plugins such as big data server, make it so that it simply pipes the data into other
  pre-existing plugins.
  - Can potentially collab with big data viewer playground
  - Pros
    - Can be created to do exactly what is desired
    - Will be more versatile with its application, able to use any processing functionality
    - Can be used by more than just VCell, but any organization that requires large sets of data to be
    processed over an internet connection
    - More than one type of image format can be served
  - Cons
    - Will require more time for development

- Mount the results
  - Have the user connect to a remote drive and mount it to their computers
  - Each mounting point corresponds to a different users account, files generated by them stored within that drive
  - Pros
    - Uses file navigation menu already in big data plugins
    - Side steps ImageJ plugin development entirely, requiring potentially less time to develop
  - Cons
    - Side steps ImageJ plugin development, may not be something we want to do
    - Managing all of these different mounting points 

- Use Bio-Formats
  - Host the resulting model image at some endpoint on the VCell website
  - Pros
    - Won't need to develop any plugins
    - Can use the processing tools offered by the bio formats plugin which is the underlying ImageJ processing plugin
  - Cons
    - Within its own documentation it's referenced as an unreliable solution
    - For large images, it would require the user to open the image in a virtual stack
    - Does not have the quality of life features that the big data plugins have

- N5-IJ Plugin
  - Using this plugin models stored within cloud storage solutions (AWS, GS) can be accessed
  - Pros
    - Plugin allows for the native I/O of ImageJ to open the model
    - Can be used with big data viewer
    - The data being hosted is not constrained to the ImageJ ecosystem
  - Cons
    - When trying to use big data viewer with the images opened it seemed to give me problems.
      - Only worked when I opened the Image already with bio-formats
    - Requires a cloud storage API for it to work

- [Xpublish](https://github.com/xpublish-community/xpublish)
  - Server side code that allows for large N-dimensional data to be served
  - Pros
    - Can execute a python script with the libraries to serve the data however desired
    - Is not constrained to the ImageJ ecosystem
    - Supports ZARR data format for API endpoint
  - Cons
    - The API endpoints are not supported by ImageJ plugins that do remote Image analysis
    most of those plugins only support S3 buckets

- ***Simple Storage Service (S3) Buckets for public storage***
  - Use the S3 buckets to store simulation results, and make them public so that people can access them
  - Pros
    - Allow the user to choose which format the file gets exported in allowing it to
    work with any remote plugin
    - Not coupled to ImageJ
    - There is a large ecosystem of applications outside of ImageJ that supports S3
  - Cons
    - Requires infrastructure to be setup, may not be too bad though

Have the simulation result within the remote repository.
Preview the data preemptively with the Big Data viewer.
Then stream the information to the user when they want to access the model within ImageJ.

  - The Fiji plugin communicates with VCell client via some internal port on the users machine

  - Plugin Process
    - Load the plugin
    - Communicates with VCell client via some [open port](https://github.com/paulricky/vcell/blob/master/vcell-imagej-helper/src/main/java/org/vcell/imagej/helper/VCellHelper.java#L86)
    - VCell client has some API that allows for the ImageJ application to speak with it
      - Seems to be that a URL is generated with the port number where VCell client API is hosted, and regex is used to parse a query to that [URL](https://github.com/paulricky/vcell/blob/master/vcell-imagej-helper/src/main/java/org/vcell/imagej/helper/VCellHelper.java#L387)
    - Requests model and sees if it is available
    - Model available then make a request
    - Treat each slice of the stack as a grid [array of pixels](https://github.com/paulricky/vcell/blob/master/vcell-imagej-helper/src/main/java/org/vcell/imagej/helper/VCellHelper.java#L161)
    and load them into an [image array](https://github.com/paulricky/vcell/blob/master/vcell-imagej-helper/src/main/java/org/vcell/imagej/plugin/VCellPlugin_Minimal.java#L1179), 
    then load that Image array [into](https://github.com/paulricky/vcell/blob/master/vcell-imagej-helper/src/main/java/org/vcell/imagej/plugin/VCellPlugin_Minimal.java#L1232) the 
    [ImgPlus object](https://imagej.nih.gov/ij/developer/api/ij/ij/ImagePlus.html), and finally show that ImagePlus object.
